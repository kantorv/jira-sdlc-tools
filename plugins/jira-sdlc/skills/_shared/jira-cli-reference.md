## 0. Auth & non-interactive flags

Check whether `JIRA_API_TOKEN` is already exported:
```
echo $JIRA_API_TOKEN
```
- **Set** → run plain `jira ...` commands, no prefix needed.
- **Empty** → fall back to the token file at `<JIRA_TOKEN_PATH>` (default
  `.jira/token.txt`, relative to the worktree root — see
  `jira-tools-plugin.env` in the project root). Prefix every command:
  ```
  JIRA_API_TOKEN="$(cat <JIRA_TOKEN_PATH>)" jira ...
  ```
  Verify with `JIRA_API_TOKEN="$(cat <JIRA_TOKEN_PATH>)" jira me` — returns
  your account email (e.g. `you@example.com`). That email string is itself
  a valid value for `-a$(jira me)` — jira-cli resolves email → accountId
  server-side, no need to extract an accountId manually.

**Non-interactive flags** — confirmed in practice that support is **not
uniform across subcommands**:
```
--no-input     # most write commands. NOT supported by `issue delete` — see §8.
--plain        # list / view ONLY. Errors "unknown flag: --plain" on
               # create and other write commands — don't add it there.
--raw          # raw JSON output (best for programmatic parsing)
```
Rule of thumb: **read** commands (`list`, `view`) → `--plain` or `--raw`.
**Write** commands (`create`, `edit`, `move`, ...) → `--no-input` + every
required flag explicit, never `--plain`, never rely on the prompt.

## 1. Issue type hierarchy

Two-level hierarchy with no `Epic` level — `Story`, `Task`, and `Bug` are
the top-level types (peers), and `Sub-task` is the only lower level:

```
Task / Story / Bug        (top-level, no parent)
 └── Sub-task              (linked to its parent via --parent)
```

| Role | Exact type name |
|---|---|
| Task | `Task` |
| Story | `Story` |
| Bug | `Bug` |
| Subtask | `Sub-task` |

Default project key: `<PROJECT-KEY>` (see `jira-tools-plugin.env` in the project root).

Note the hyphen in `Sub-task` — pass it quoted in commands: `-t"Sub-task"`.

## 2. Creating issues

**Top-level Task / Story / Bug:**
```
jira issue create -t<Task|Story|Bug> -s"<summary>" -b"<description>" --no-input
```

**Sub-task, attached to a parent Task/Story/Bug** (the parent must NOT
itself be a `Sub-task` — see the type-check in §3):
```
jira issue create -t"Sub-task" -s"<summary>" -b"<description>" -P<PARENT-KEY> --no-input
```

⚠️ Always quote the type as `-t"Sub-task"`. A bare unquoted `-tSub-task`
parses incorrectly (reads as `-t Sub`) rather than erroring clearly.

⚠️ **Inline `-b"<long description>"` has been observed to hang for ~2
minutes** on anything beyond a couple of sentences. For real descriptions,
write the body to a file and load it with `--template` instead — note this
is `--template`, there is no documented short alias:
```
cat > /tmp/issue-body.md <<'EOF'
...markdown body...
EOF

jira issue create -t Task -s "<summary>" --template /tmp/issue-body.md \
  -a "$(jira me)" --no-input
```
`--template -` also reads the body from stdin, if you'd rather pipe it
(same as `echo "..." | jira issue create ...` below).

Other useful create flags:
```
-y<Priority>          # e.g. -yHigh
-l<label>             # repeatable: -lbackend -lurgent
-a<assignee>          # use $(jira me) for self
-C<component>
--fix-version <v>
--custom <field>=<value>   # custom fields — see repo discussion #346
```

Description supports Markdown directly, or load from stdin:
```
echo "Generated by Claude Code agent.\n\n<details>" | jira issue create -s"<summary>" -tTask --no-input
```

**Splitting a task into parallel Sub-tasks (the actual workflow):**
1. Create the parent Task/Story/Bug first, capture its returned key (parse with `--raw`).
2. For each genuinely independent piece of work, create a `Sub-task` with `-P<PARENT-KEY>`.
3. Don't create Sub-tasks for purely sequential steps — only for parallelizable work.
4. The parent `Task`/`Story`/`Bug` is the top of the hierarchy — there's
   no grouping above it.

## 3. Reading / listing issues

```
jira issue list --plain                          # recent issues, current project
jira issue list -a$(jira me) --plain             # assigned to me
jira issue list -s"<STATUS_TODO>" --plain        # filter by status
jira issue list -t<Type> --plain                 # by issue type
jira issue list -q "summary ~ keyword" --plain   # raw JQL filter
jira issue view <KEY>                            # full detail incl. comments/links
jira issue view <KEY> --raw                      # JSON, for parsing
```

**Checking an issue's type and parent** (needed by SKILL.md's branch-based
parent detection — a `Sub-task` can't itself be a parent, so the skill walks
up to its own parent first). Confirmed in practice: Jira rejects a `Sub-task`
as a `-P` target with `400 Bad Request: Parent issue ... can not be sub-task`.
```
jira issue view <KEY> --raw
# CHECK exact field paths against real output once, expected:
#   .fields.issuetype.name   — e.g. "Task", "Story", "Bug", "Sub-task"
#   .fields.parent.key       — only present when <KEY> is itself a Sub-task
```

Sprints (independent of issue hierarchy):
```
jira sprint list --current           # current sprint's issues
jira sprint list --current -a$(jira me)
```

## 6. Comments & worklogs

### Add a comment

**Single-line (simple text):**
```bash
jira issue comment add <KEY> "Single-line comment text"
```

**Multi-line / markdown / backticks (safest — heredoc to `--template -`):**
```bash
cat <<'EOF' | jira issue comment add <KEY> --template -
Multi-line comment with **markdown** and `inline code`.
Line 2 of the comment.
Line 3 of the comment.
EOF
```

**Dynamic / single-line via `echo` pipe (good for templated strings):**
```bash
echo "Comment from stdin" | jira issue comment add <KEY> --template -
```

**Bash dollar-quoted string (inline newlines without heredoc):**
```bash
jira issue comment add <KEY> $'Line 1\nLine 2\nLine 3'
```

**From a file (reusable template / long report):**
```bash
jira issue comment add <KEY> --template /path/to/comment.md
```

#### ⚠️ Important notes for comments
- **There is no `-m` flag** — `jira issue comment add KEY -m "text"` does not exist. The positional argument works exactly as in `jira issue create`.
- **Backticks inside double-quoted strings are dangerous in a raw shell** because bash treats `` ` `` as command substitution. The heredoc or file methods are safest for any comment containing backticks, markdown, or newlines.
- **The `--template -` form reads stdin** — explicit `echo "..." | jira issue comment add <KEY> --template -` is the canonical multi-line pattern used across all skills.
- **Pipe without `--template` also works** — `echo "..." | jira issue comment add <KEY>` is accepted as a shorthand when stdin is a pipe.

### Worklog

```bash
jira issue worklog add <KEY> "1h 30m" --comment "<note>" --no-input
```

## 7. Git workflow — branch convention

**Decision rule:** every change goes on its own branch using the
feature/hotfix convention below — no matter how small. There's no
"small enough to commit straight to the working branch" shortcut in
this toolkit.

**When running as `jira-task-executor`**: the assigner pre-creates the
branch and worktree for every leaf issue, so you're resuming or
creating that branch per the skill's step 2 — never committing straight
to the base or parent branch.

GitHub-for-Jira links a branch to an issue purely by finding the **issue key inside
the branch name** — no API call required either.

```
git checkout -b <feature|hotfix>/<ISSUE-KEY>-<short-description>
git push -u origin <feature|hotfix>/<ISSUE-KEY>-<short-description>
```

Once pushed, the branch shows up automatically in the issue's "Development" panel.

**Confirmed naming convention — use exactly this, no other variants:**
```
feature/<ISSUE-KEY>-<slugified-title>      # Task / Story (and Sub-tasks under them)
hotfix/<ISSUE-KEY>-<slugified-title>       # Bug (and Sub-tasks under them)
```

Examples:
```
feature/PROJ-123-add-rate-limit
hotfix/PROJ-345-fix-null-pointer-on-login
```

Rule for picking the prefix: use the **type of the top-level parent issue**
(Task/Story → `feature/`, Bug → `hotfix/`). A Sub-task takes its prefix from
its parent's type, not its own — a Sub-task under a Bug is still `hotfix/...`.

Slugify the title: lowercase, spaces → hyphens, strip punctuation.
`"Fix null pointer on login!"` → `fix-null-pointer-on-login`.

## 8. Destructive / risky commands — use with care

```
jira issue delete <KEY>              # interactive y/N prompt — no flag skips it
jira issue delete <KEY> --cascade    # deletes Sub-tasks too
```

⚠️ `delete` does **not** support `--no-input` — confirmed it errors with
`unknown flag: --no-input`. There's no flag-based way to skip the
confirmation prompt, so the agent can't run this reliably non-interactively.

Agent rule: never run `delete` without the user explicitly asking for that
issue to be deleted in that message — and even then, **don't try to
automate past the interactive prompt.** Hand the user the exact,
ready-to-paste command instead and let them run it themselves:
```
JIRA_API_TOKEN="$(cat <JIRA_TOKEN_PATH>)" jira issue delete PROJ-277
```
This matters especially for any throwaway/smoke-test issues created while
diagnosing a skill in this plugin — surface the delete command in the final
report rather than deleting them automatically, even though they were
created in the same session.

`create`, `edit`, `move`, `link`, `comment`, `worklog` are reversible/
low-risk and fine to run once confirmed in the workflow.

## 10. gh tool usage examples

**PR creation** — always include `--label` for semver:
```
gh pr create --base development --title "PROJ-123: fix off-by-one" --body "..." --label patch
```
Label picks the version bump: `patch` (fixes/internal), `minor` (features), `major` (breaking).
Labels must exist in the repo — check with `gh api repos/<org>/<repo>/labels --jq '.[].name'`.
(Label names shown are the `<SEMVER_LABELS>` default from `jira-tools-plugin.env` — adjust if yours differ.)

**Other gh commands:**
```
gh pr view --json files         
gh pr view --json mergedAt        
gh pr merge --squash + git push origin --delete <branch>
```
